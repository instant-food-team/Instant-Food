# ARCHITECTURE_RULES

## 1. 最终交付形态
《拍立食》的最终正式交付形态固定为：

`微信小程序`

当前 `frontend/prototype/Chinese/` 中的页面只是 H5 原型，不再被视为最终正式技术栈。

## 2. 固定架构基线
当前统一采用最小化分层架构：

`微信小程序前端 -> 单体后端 API -> 关系型数据库 + 对象存储`

规则：
- 小程序前端不直连数据库。
- 小程序前端不直接调用第三方 AI 服务。
- 所有登录、上传、识别、生成、归档、社区读取、Shop 配置都由后端统一提供接口。

## 3. 技术栈口径

### 前端正式技术栈
- 原生微信小程序
- `WXML`
- `WXSS`
- `JavaScript`
- 工程化阶段建议启用 `TypeScript`

### 后端推荐技术栈
- `FastAPI`
- `Python`

### 数据层
- 单一关系型数据库
- 推荐 `MySQL 8`
- 或 `PostgreSQL`

### 存储层
- 对象存储

禁止项：
- 不再把 React/Vite/Tailwind 写成最终正式前端栈。
- 不允许前后端各自维护不同接口字段。
- 不允许擅自拆微服务、消息队列、BFF、网关、事件总线。
- 不允许因为社区或 Shop 的展示需求重做整套技术架构。

## 4. 团队协作边界
- 前端负责页面、交互、状态、mock、接口适配、小程序端实现。
- 后端负责登录、上传、识别、生成、归档、设置、社区读取、Shop 配置。
- 数据库同学负责表结构、迁移、初始化数据、数据字典。
- 发布同学负责 AppID、域名白名单、web-view 配置、提审发布。

边界解释：
- `社区` 当前不是完整社区系统，只是展示别人分享内容的 UI 和只读数据层。
- `Shop` 当前不是电商域，只是第三方平台导购壳。
- 任何人都不得把这两个模块自行扩成真实社区系统或真实商城系统。

## 5. 目录约束
- `frontend/prototype/`：当前 H5 原型
- `frontend/assets/`：原型直接使用的素材
- `frontend/miniprogram/`：正式小程序前端代码
- `backend/api/`：后端接口实现
- `backend/database/`：数据库脚本与迁移
- `docs/architecture/`：架构与协作文档
- `docs/planning/`：范围与任务板
- `docs/backend/`：接口与数据模型
- `docs/ui/`：UI 风格与 UI 协作提示词
- `tests/visual/`：测试截图与验收图
- `tmp/browser/`：临时浏览器目录

## 6. 当前固定交互规则
以下交互规则已经收口，不允许自由发挥：
- 底部第二栏固定为 `社区`。
- 底部中间 `+` 固定直达 `拍摄.html`。
- 不再保留文字输入、语音输入或工具弹层。
- 拍摄页主 CTA 固定为 `确认拍摄`，点击后进入分子重构确认页。
- 分子重构确认页中的主要按钮必须具备真实前端交互，不允许保留“只可看不可点”的假按钮。
- 分子重构确认页底部主 CTA 先进入加载过渡页的确认菜单态，再由确认菜单进入加载并开始生成。
- 首页当前不保留 `最近档案` 和 `身份摘要` 模块。
- 食材确认、加载过渡、结果页兜底文案里的数量展示优先使用 `块 / 片 / 颗 / 瓣` 这类口语单位。

## 7. 小程序特有限制
- 登录必须走微信登录。
- 正式前端不使用 `window`、`localStorage`、`sessionStorage`。
- 正式前端不使用浏览器任意外链跳转。
- Shop 模块必须在 `web-view / 合作小程序 / 复制链接引导` 中选择正式落地方式。

## 8. 社区模块规则
社区当前只允许做这些：
- 共享菜谱卡片。
- 社区列表壳或详情壳。
- 作者信息、封面图、摘要、食材预览。
- “去买同款食材”或等价 CTA。
- 基于假数据的浏览、筛选、详情展示。

社区当前禁止做这些：
- 评论
- 点赞
- 收藏
- 关注
- 私信
- 推荐流
- 排行榜算法
- 审核后台
- 举报处理
- 真实内容发布链路

## 9. Shop 模块规则
Shop 当前只允许做这些：
- 从社区内容进入 Shop。
- 展示第三方平台入口，例如盒马、美团、叮咚买菜。
- 展示食材包或食材清单的前端信息。
- 点击后跳转到外部链接或占位跳转地址。

Shop 当前禁止做这些：
- 购物车
- 下单
- 支付
- 地址管理
- 配送状态
- 库存同步
- 价格实时拉取
- 平台登录绑定
- 第三方真实 API 对接

## 10. 数据与接口规则
当前优先使用假数据闭环，再决定是否接真实接口。

允许预留的社区字段：
- `shared_recipe_id`
- `source_archive_id`
- `title`
- `cover_image_url`
- `summary`
- `ingredient_preview`
- `author_name`
- `author_avatar_url`
- `shared_at`
- `shop_entry_id`

允许预留的 Shop 字段：
- `shop_entry_id`
- `shared_recipe_id`
- `ingredient_bundle_name`
- `ingredient_list`
- `platform_name`
- `platform_type`
- `redirect_url`
- `is_mock`

规则：
- 可以定义类型、mock、假接口适配层。
- 不要因为“以后可能会用到”就先把真实业务接口做完。
- 如需预留接口，只保留最小 contract，不实现真实业务逻辑。

## 11. Codex 禁止自由发挥事项
后续 Codex 明确禁止做以下行为：
- 擅自引入复杂新架构。
- 擅自把社区做成真实社区系统。
- 擅自把 Shop 做成真实商城。
- 实现评论、点赞、关注、推荐、审核后台。
- 恢复旧的文字/语音输入分支。
- 因为局部页面想“更完整”而大范围重构无关模块。
- 擅自改动主链路顺序。

## 12. 改动优先级
所有实现决策统一按以下优先级排序：
1. 主链路是否能走通。
2. Nexus 是否仍然是首页中枢。
3. 档案馆是否稳定承接归档结果。
4. 社区 UI 是否表达清楚。
5. Shop 外跳导购是否成立。

如果某个改动会伤害更高优先级模块，就不做。
