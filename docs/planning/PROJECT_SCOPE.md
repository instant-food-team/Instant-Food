# PROJECT_SCOPE

## 1. 项目定义
《拍立食》当前不是传统菜谱 App，不是完整社区产品，也不是电商平台。

当前项目定义为一个最终落地到微信小程序的“剩余食材重构体验”产品，核心由三层组成：
- 主链路：用户输入或拍摄剩余食材，经过识别、确认、生成，得到一道重构菜谱并保存归档。
- 资产层：档案馆沉淀用户自己的历史作品，承担保存、回看、展示的职责。
- 展示支线：Nexus 中的社区内容展示，以及从社区内容跳转到第三方买菜平台的 Shop 导购壳。

当前执行一律以 `docs/` 目录下的治理文档为准。

## 2. 当前阶段
当前阶段固定为：

`H5 原型收口 + 微信小程序实现对齐 + MVP 主链路 + 社区 UI 前端 + Shop 导购壳`

这句话的含义必须统一理解：
- `frontend/prototype/Chinese/` 是当前视觉和交互锚点。
- 正式产品最终要落在微信小程序，不再把 H5 当正式前端。
- `MVP 主链路` 要真能跑通。
- `社区 UI 前端` 只做展示，不做真实社区业务。
- `Shop 导购壳` 只做导购入口，不做真实商城交易。

## 3. 当前固定主链路
当前主链路固定为：

`微信登录 / 首页 Nexus / 拍摄识别 / 分子重构确认 / 加载过渡 / 结果页 / 保存归档 / 档案馆`

这条链路的执行要求如下：
- Nexus 是首页中枢，负责承接主链路入口和社区预览入口。
- 底部第二栏固定为 `社区`，不再使用 `商城` 作为主导航名称。
- 底部中间 `+` 固定直达 `拍摄.html`。
- 不再保留文字输入、语音输入或工具输入弹层。
- 拍摄页主 CTA 固定文案为 `确认拍摄`，点击后先进入 `分子重构台.html`。
- 分子重构确认页中的交互控件需要可操作，至少包括食材数量调整、保留/剔除、技法选择、厨具选择、味觉选择。
- 分子重构确认页底部主 CTA 先进入 `加载过渡.html` 的确认菜单态，再由确认菜单进入加载生成。
- 分子重构确认页、加载过渡页、结果页兜底文案中的食材数量，优先使用 `块 / 片 / 颗 / 瓣` 这类口语化家庭单位，不再优先显示 `g` 重量。
- 首页当前不再保留 `最近档案` 和 `身份摘要` 作为首页核心模块。
- 小程序正式实现时，不再依赖浏览器的 `sessionStorage`、`window.location` 等 H5 能力。

## 4. 当前必须做什么

### 4.1 主链路必须可演示
- 用户能从登录进入 Nexus。
- 用户能从 Nexus 直接进入拍摄。
- 用户能完成食材确认，再进入加载过渡和结果页。
- 用户能保存结果，并在档案馆回看历史记录。

### 4.2 社区当前必须做到的范围
- 在 Nexus 或社区页展示别人分享的黑暗料理内容。
- 展示内容可以来自假数据、档案衍生数据或前端静态数据。
- 社区内容只承担“种草”和“引导去买同款食材”的作用。
- 可以有社区列表壳、详情壳、作者信息、摘要、封面图、食材预览。

### 4.3 Shop 当前必须做到的范围
- 用户能从社区内容进入 Shop。
- Shop 只展示第三方平台入口，例如盒马、美团、叮咚买菜等。
- Shop 的平台卡片本质是导购入口，不承担下单、支付、库存、价格同步等功能。
- 小程序正式落地时，导购入口要基于平台限制选择 `web-view / 合作小程序 / 复制链接引导` 方案。

## 5. 当前明确不做什么
- 不做真实社区后端。
- 不做评论、点赞、收藏、关注、私信。
- 不做推荐流、热榜算法、个性化分发。
- 不做社区审核后台、举报处理、内容治理系统。
- 不做复杂通知系统、消息中心、推送编排。
- 不做真实商城能力，包括购物车、地址、订单、支付、库存、优惠券。
- 不做第三方买菜平台真实 API 对接。
- 不做额外输入分支，不恢复文字/语音交互入口。

## 6. 当前成功定义
只有同时满足以下条件，当前阶段才算完成：
- 主链路从登录到保存归档能走通。
- 最终技术方向已经统一为“微信小程序前端 + 后端 API”。
- Nexus 能承接首页中枢职责。
- 社区分享 UI 可见且逻辑自洽。
- 用户能从社区内容进入 Shop。
- Shop 能展示第三方平台跳转入口。
- 整体实现仍然保持 `社区前端化 / Shop 导购化 / 主链路优先`。
